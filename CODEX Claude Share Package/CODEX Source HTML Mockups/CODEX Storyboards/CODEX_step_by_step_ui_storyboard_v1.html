PG Testing + Audit desktop interface, step by step

This storyboard turns the MVP into a concrete user experience. The tester side stays fast and low-friction; the PG side becomes a calm approval queue where every AI-created issue has evidence, transcript context, and a human-reviewed response.

One obvious next actionEach state makes the next action visually dominant while secondary actions stay quiet.
Evidence near decisionsTranscript, screenshots, and issue text stay attached to the decision point.
Fast captureRegion capture saves immediately. Review is optional from a toast.
Human approvalAI drafts and groups issues, but PG approves before anything is sent.
Searchable archiveClosed issues remain findable with original evidence and response history.

Step 3

Shift+F12 enters region capture mode

The screen dims, a see-through rectangle reveals the exact captured area, and live dimensions follow the cursor.
State: selecting region
Drag to capture region. Esc to cancel.
305 x 172 px

User intent

The tester noticed something specific and wants evidence quickly.

Design details

  • No file picker or naming prompt.
  • Peach outline matches the app's active state language.
  • Esc is visible so the mode feels reversible.
Capture first, optional review second.
Step 4

Capture succeeds with a toast and optional review

The file saves immediately. A toast confirms success and gives a temporary path into review without interrupting the test.

User intent

Most captures need no follow-up. The review dialog is there only when the tester wants it.

Design details

  • Success toast is visible but does not block the workflow.
  • Review has only three decisions: Save, Discard, Re-capture.
  • Discard must remove the file and update session results.
Review is a safety valve, not a required step.

User intent

PG wants concrete, evidence-backed issues, not raw recordings.

Design details

  • Priority is visible in the first column.

Step 9

Closed issues become a searchable audit archive

Every resolved issue keeps original evidence, transcript anchors, response history, and status so future repeats can be found quickly.

User intent

When an issue repeats, PG should find the old evidence and response in seconds.

Design details

  • Search is phrase-first, with filters available but not mandatory.
  • Results show preview, title, date, category, status, and priority.
  • The archive is a knowledge base, not a storage dump.
Searchable audit integrity is MVP behavior, not extra polish.
