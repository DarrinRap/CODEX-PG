Shell mockup v1 — Library module — designer's notes

This is iteration 1 of the v4.0 UI shell, scoped to the Library module in patient-selected state at 1440×900. Everything below is open for critique. Nothing is locked.

What's in this mockup

  • Module tabs (Library / Arrange / Present) top-left with keyboard hints. Active tab underlined in the mode-color token.
  • Back / Forward nav buttons (bug #116). Separate from edit undo.
  • Patient list left pane with search, avatar, ID/sex/year, image count.
  • Thumbnail grid center with type badges (PA/BW/Photo), date+sequence overlay.
  • Filmstrip persistent below the grid — all patient images across visits.
  • Right pane with 5 collapsible sections: Patient Info, Filter, Sort, Selected Image, Notes & Tags (shown collapsed).
  • Status bar with mode label in the mode-color, active filters, sort, DB context.
  • Import / Export icons corrected: ↓ for import, ↑ for export (#115).
  • Labeled buttons everywhere — no icon-only buttons in viewport controls (#119).

Mode zone-color (UX §2.2)

The LIBRARY label in the status bar uses the mode-library token (blue-gray). The active module tab underline matches. Arrange will use orange, Present will use green. Canvas-surround color shift is subtle here; plan is Library stays near-black, Develop-equivalent (Image Editor within Arrange) shifts to #2a2a2a.

Decisions I made without asking

  • Filmstrip is always visible in Library (not just during image edit). Gives persistent "where am I in this patient's history" context. Close button in the header lets it hide if that's wrong.
  • Import / Export / Present as the three global action buttons top-right. Present is accented because it's the differentiated feature. Open to demoting it if that's too loud.
  • Patient avatars are initials-based — no real photos. Simpler and more appropriate for a clinical app.
  • Sub-sections have disclosure arrows and persist state (per UX §1.6). Notes & Tags shown collapsed to demonstrate.

What I'm unsure about

  • Filmstrip vs. selection panel competition. Both show image-sequence context. If the filmstrip is always visible, does the right-pane "Selected Image" section duplicate information? Lean: keep both — filmstrip for navigation, selection panel for metadata inspection.
  • Left-pane patient count. "247 patients" feels small for a mature practice; might be too big for a new install. If there's a known ballpark for Darrin's DB, I can calibrate.
  • Module tab position. Top-left (shown) vs. top-center (Lightroom convention). Top-left gives more room for global actions on the right. Top-center is more visually balanced.
  • Action bar density. Import / Export / Present may deserve separate breathing room from the tabs. Currently they're both in the 44px title bar.
  • Dark shade. Canvas is #1a1a1a, panes #222228, chrome #17171c. Slightly more contrast than v3. Push darker or lighter?

What's deliberately NOT in this mockup

  • Empty-state (no patient selected) — separate mockup, addresses bug #114
  • Arrange module canvas — Month 2 work, placeholder in Week 1 shell
  • Present module — Month 2 work
  • The right-panel collapsible persistence across sessions (UX §1.6) — behavior, not visual
  • Loupe / Develop sub-mode within Library — if Darrin wants it; UX §2.3 had it, plan consolidates to separate Arrange module
  • Comparison view (right button in view controls is present but unstated as a mode)

Questions for you

  • Is Library really the right first-module? UX doc said Library/Develop/Slideshow/Presentation; plan says Library/Arrange/Present. Is "Develop" absorbed by Arrange's image-editor sub-mode, or is it its own thing?
  • Does the thumbnail metadata show what you want? Currently: date + sequence-number. Could be: region, tooth number, edit count, file size.
  • Patient list ordering — should "most recent" pin to the top, or is alphabetical right?
  • Should Present be a module tab OR a button (as shown in the top-right) — not both? Currently both exist. Lean: tab=destination, button=action ("Start presentation now").
