Pane-level FAIL panel — mockup v2

Phase B of bug #87. Single-expected steps: replace the action row (PASS / FAIL) with an inline FAIL panel when the user taps FAIL, so the user can type a note and commit without leaving the pane. Visual language reuses shipped CHECKLIST_VIEW_QSS object names verbatim — #checklistFailPanel, #checklistFailPanelEdit, #checklistBtnPanelCancel, #checklistBtnPanelSave.

v2 fix vs v1: state 2 now shows sample text typed into the textarea so SAVE FAIL renders in its enabled red state (matches shipped #checklistBtnPanelSave — color #d46a6a, border rgba(212,106,106,0.35)). v1 accidentally drew the disabled (gray) state because the textarea was empty. A dedicated swatch board at the bottom shows every button state.
Three states

State 1 — Default. Body + EXPECTED + action row (PASS / FAIL). Unchanged from v3.75.6.

State 2 — FAIL panel open. Action row replaced with an inline panel: red-tinted container, "DESCRIBE WHAT FAILED" header, multi-line textarea, CANCEL + SAVE FAIL button row. Bottom-of-pane Cancel / Pause / ← Back row stays visible (pane-level actions).

State 3 — Post-save. Result written to results_latest.json, auto-advance to the next step (or to summary on last step). Identical to PASS advance.

State transitions

State 1 → State 2 — User opens the FAIL panel

  1. User taps FAIL (or presses F).
  2. _on_fail detects single-expected step, sets self._fail_panel_open = True.
  3. Action row (PASS + FAIL buttons) hidden.
  4. FAIL panel mounted above the bottom row: header, textarea, CANCEL + SAVE FAIL buttons.
  5. Textarea receives focus. SAVE FAIL starts disabled.
  6. Body + EXPECTED remain visible above — _body_filler still absorbs leftover space.
  7. Bottom row (Cancel / Pause / ← Back) stays visible. ← Back is enabled only if step_index > 0.

State 2 → State 1 — User cancels the panel

  1. User taps CANCEL or presses Esc.
  2. _close_fail_panel sets self._fail_panel_open = False.
  3. Panel torn down (widgets setParent(None) + deleteLater(), matching _ChecklistRow._clear_extra).
  4. Action row re-shown (PASS + FAIL).
  5. No result written. No timestamp. Step state unchanged.
  6. Focus returns to PASS button (matches _render_step's default).

State 2 → State 3 — User submits the FAIL

  1. Textarea non-empty (after .strip()). User taps SAVE FAIL or presses Enter.
  2. _submit_fail_panel reads the text, calls self._results_writer.record_single(step_n, test_id, title, outcome="FAIL", note=text).
  3. Panel torn down. self._fail_panel_open = False.
  4. If last step: _render_summary. Else: _render_step(self._step_index + 1).
  5. Mirrors _on_pass's post-record advance exactly — same branching, same mark_completed on summary.

Invariants the implementation must preserve

  • Checklist path untouched. _on_fail's existing if self._is_checklist_step(): branch that routes to self._checklist_view.handle_fail() stays first. Phase B only changes the single-expected branch below it — the old stub dialog goes away.
  • Schema v1 files work unchanged. Phase A's verification plan is schema v1 — no fail_chips field. The "If fail_chips absent" spec §5.2 variant is the ONLY one Phase B ships. Chips are deferred.
  • Paused state propagates. When self._paused becomes true while the panel is open: textarea and both buttons disable. When unpaused: re-enable. Mirrors _ChecklistRow.set_paused.
  • Back in state 2 cancels the panel. If the FAIL panel is open, _on_back (Shift+Enter) closes it instead of navigating to the previous step. Mirrors ChecklistStepView.handle_back's open-panel-first behavior.
  • Esc in state 2 cancels the panel. If the FAIL panel is open, _on_cancel (Esc) closes it instead of triggering the cancel-session prompt. Mirrors ChecklistStepView.handle_cancel.
  • Results writer guarded. record_single(..., outcome="FAIL", note=text) call wrapped in try/except; failure logs a warning via log.warning but does not block the advance. Same pattern as Phase A's _on_pass.
  • No screenshot write. Spec §5.8 auto-screenshot is out of scope (v3.68). screenshot field stays null per results_writer's default. No capture call in Phase B.
  • No fail_chips, no report hint. Spec §5.2 covers chips and the report field pre-fill. Phase B deliberately ships only the plain-textarea variant. Deferred to later phase.
  • QSS reused, not duplicated. Reuse existing #checklistFailPanel, #checklistFailPanelLabel, #checklistFailPanelEdit, #checklistBtnPanelCancel, #checklistBtnPanelSave object names so CHECKLIST_VIEW_QSS styles this panel verbatim. No new styles.py edit.
  • Reuse _FailNoteEdit class. Already exists at module level with Enter / Shift+Enter / Esc semantics correct. New code imports nothing new; constructs a _FailNoteEdit(parent).
